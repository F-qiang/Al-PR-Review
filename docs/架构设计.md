# 架构设计

## 1. 背景

开发者在 Review Pull Request 时需要阅读大量 diff，容易遗漏安全风险与逻辑问题。本系统通过 GitHub API 自动获取 PR 变更，结合规则引擎与 LLM 生成结构化评审报告。

## 2. 系统架构

```
用户 → Next.js 前端 → FastAPI 后端 → GitHub API
                         ↓
                    LLM 推理 + 规则引擎
                         ↓
                    SQLite +（扩展）七牛云 OSS
```

## 3. 模型选择

| 方案 | 优点 | 缺点 |
|------|------|------|
| DeepSeek | 成本低、中文好、OpenAI 兼容 | 需网络 |
| 通义千问 | 国内稳定 | 按量计费 |
| GPT-4o | 分析质量高 | 成本较高 |

**当前选型**：DeepSeek（`deepseek-chat`），通过 OpenAI 兼容接口接入，便于后续切换模型。

## 4. 上下文获取策略

1. 拉取 PR 元数据（title、body、stats）
2. 分页获取 files changed 及 patch
3. 按文件类型优先级排序（代码文件优先）
4. 总 patch 超过约 48K 字符时截断，优先保留变更量大的代码文件
5. 未来扩展：拉取关联文件（如被修改函数的调用方）做 RAG

## 5. 误报与漏报控制

- **规则预筛**：硬编码密钥、SQL 拼接、eval、TLS 验证关闭、宽泛异常捕获
- **LLM 约束**：Prompt 要求基于 diff 分析、不确定项标 low severity
- **结果合并**：规则与 LLM 结果去重，按 severity 排序
- **未来**：Semgrep 静态规则 + 多模型交叉验证

## 6. 扩展方向

- GitHub App：PR 打开时 Webhook 自动触发 Review 并评论
- 七牛云 Kodo：存储 diff 快照与 PDF 报告
- CI 集成：GitHub Action 步骤
- 大 PR：按文件并行分析后 merge 结果
